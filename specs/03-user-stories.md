## US-001 — Map structured MLS state errors to safe user-facing messages

**Adopted from Issue #24** — follow-up of PR #23 (MLS state envelope),
residual item 3 of `docs/security/2026-07-12-review-mls-state-envelope.md`.

The unlock UI currently shows raw `err.message`. Goal:

- map stable `MlsStateError` codes (`MLS_STATE_*`) to user-facing messages;
- technical details only in development logs;
- never any payload or MLS material (already guaranteed by the library
  contract; the UI must preserve it);
- controlled exposure of the structured recovery `actions` carried by
  `MLS_STATE_OPENMLS_INCOMPATIBLE` (reopen with the writing build / wait for
  a migrating build / explicit factory reset as last resort).

Not a security fix (no sensitive data in current messages — asserted by
tests); it is a usability requirement for the fail-closed behaviour
introduced by the envelope.

### Acceptance criteria

- DATO un errore `MlsStateError` con codice stabile `MLS_STATE_*`
  QUANDO la unlock UI lo presenta all'utente
  ALLORA mostra il messaggio mappato per quel codice, mai il raw
  `err.message`.
- DATO un errore `MLS_STATE_OPENMLS_INCOMPATIBLE` con `actions` strutturate
  QUANDO la UI propone il recupero
  ALLORA espone solo le azioni previste (reopen con la build scrivente /
  attendere una build migrante / factory reset esplicito come ultima
  risorsa), nell'ordine di minor distruttività.
- DATO un codice errore non riconosciuto dalla mappa
  QUANDO la UI lo riceve
  ALLORA mostra un messaggio generico sicuro e registra i dettagli tecnici
  solo nei log di sviluppo.
- DATO qualunque percorso di errore della unlock UI
  QUANDO il messaggio viene composto
  ALLORA nessun payload né materiale MLS compare nel testo mostrato
  (proprietà preservata dal contratto della libreria, verificata da test).

## US-002 — Replace spread-based Uint8Array→Base64 conversion on the persistence path

**Adopted from Issue #25** — follow-up of PR #23, residual item 2 of
`docs/security/2026-07-12-review-mls-state-envelope.md`.

`bytesToBase64` (`styx-js/src/utils.js`) uses
`btoa(String.fromCharCode(...bytes))`: the argument spread hits the engine's
argument-count limit (RangeError) far below the envelope's 16 MiB parser cap,
so the practical size limit of `_persistMls` is the helper, not the parser.

To be resolved in Blocco 3 with one of:

- chunked conversion (fixed-size `String.fromCharCode` windows);
- streaming conversion;
- **preferred:** removing Base64 from the persistence path entirely — the
  IndexedDB vault stores binary values natively.

No production change during the spikes.

### Acceptance criteria

- DATO uno stato MLS serializzato di dimensione prossima al cap del parser
  QUANDO `_persistMls` lo persiste
  ALLORA la conversione non solleva `RangeError` per limite di argomenti e
  il limite pratico torna a essere il cap del parser, non l'helper.
- DATO il percorso di persistenza scelto in Blocco 3
  QUANDO la soluzione è integrata
  ALLORA una delle tre opzioni dichiarate è implementata e la scelta è
  registrata (preferita: eliminazione di Base64 con IndexedDB nativo).
- DATO il periodo degli spike
  QUANDO si lavora su questa story
  ALLORA nessun cambiamento raggiunge il percorso di produzione prima della
  decisione di Blocco 3.
- DATO un valore che ecceda comunque i limiti del backend
  QUANDO la persistenza fallisce
  ALLORA il fallimento resta fail-closed e non distruttivo (proprietà della
  story US-004 preservata).

## US-003 — Allowlist for MlsStateError.details fields

**Adopted from Issue #26** — follow-up of PR #23, residual item 1 of
`docs/security/2026-07-12-review-mls-state-envelope.md`.

`details.causeMessage` in `MLS_STATE_MIGRATION_FAILED` /
`MLS_STATE_RESTORE_FAILED` forwards the underlying runtime message and is the
only non-allowlisted channel in the structured errors. The hardened WASM
boundary returns clean errors today (verified by the no-trap tests), but the
contract should be explicit.

Define an allowlist of publishable `details` fields, e.g.: error code;
envelope version; storage schema version; saved and current OpenMLS revision;
saved and current artifact digest; saved and current ciphersuite; suggested
actions.

`causeMessage` must not be propagated automatically: keep the underlying
error for development logs only (or map it to a stable sub-code).

### Acceptance criteria

- DATO un `MlsStateError` con `details`
  QUANDO l'errore attraversa il confine verso la UI o la telemetria
  ALLORA solo i campi dell'allowlist esplicita sono pubblicabili.
- DATO un errore `MLS_STATE_MIGRATION_FAILED` o `MLS_STATE_RESTORE_FAILED`
  QUANDO il runtime sottostante fornisce un messaggio causa
  ALLORA `causeMessage` non è propagato automaticamente: resta nei log di
  sviluppo o è mappato a un sub-codice stabile.
- DATO l'allowlist definita
  QUANDO un nuovo campo viene aggiunto a `details`
  ALLORA il campo è non-pubblicabile finché non è esplicitamente aggiunto
  all'allowlist (default-deny).
- DATO i no-trap test esistenti
  QUANDO l'allowlist è introdotta
  ALLORA i test continuano a passare e un test nuovo asserisce il contratto
  esplicito dell'allowlist.

## US-004 — Document the operational limits of localStorage vs the 16 MiB parser cap

**Adopted from Issue #27** — operational-limit record requested at the
Fase D gate of PR #23.

Facts to state explicitly in the storage documentation (and to keep true in
code):

- `MAX_PAYLOAD_BYTES = 16 MiB` is the **defensive cap of the envelope
  parser**, not a guaranteed capacity of the current backend;
- 16 MiB of binary become >21 MiB in Base64, plus JSON overhead — well
  beyond typical localStorage quotas;
- browsers apply **different quotas** (commonly ~5–10 MB per origin for
  localStorage), so the practical ceiling is browser-dependent and much
  lower than the cap;
- the Blocco 3 IndexedDB vault replaces this operational limit (binary
  values, larger quotas, `navigator.storage.estimate()`);
- a quota error must remain **fail-closed and non-destructive**: today
  `QuotaExceededError` during migration keeps the legacy value + backup
  intact and raises `MLS_STATE_MIGRATION_FAILED` (tested); any future write
  path must preserve this property.

### Acceptance criteria

- DATO la documentazione dello storage
  QUANDO la story è completa
  ALLORA i cinque fatti elencati sopra sono dichiarati esplicitamente e
  coerenti con il codice.
- DATO un `QuotaExceededError` durante la migrazione
  QUANDO il fallimento avviene
  ALLORA il valore legacy e il backup restano intatti e viene sollevato
  `MLS_STATE_MIGRATION_FAILED` (proprietà già testata, da preservare).
- DATO un futuro percorso di scrittura dello storage
  QUANDO viene introdotto
  ALLORA la proprietà fail-closed e non distruttiva su quota è preservata e
  la documentazione resta vera.
