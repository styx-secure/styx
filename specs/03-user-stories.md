## US-001 — Map structured MLS state errors to safe user-facing messages

**Adopted from Issue #24** — follow-up of PR #23 (MLS state envelope),
residual item 3 of `docs/security/2026-07-12-review-mls-state-envelope.md`.

The unlock UI currently shows raw `err.message`. Goal:

- map stable `MlsStateError` codes (`MLS_STATE_*`) to user-facing messages;
- technical details only in development logs;
- never any payload or MLS material (already guaranteed by the library
  contract; the UI must preserve it);
- controlled exposure of the structured recovery `actions` carried by
  `MLS_STATE_OPENMLS_INCOMPATIBLE` (reopen with the writing build / wait for
  a migrating build / explicit factory reset as last resort).

Not a security fix (no sensitive data in current messages — asserted by
tests); it is a usability requirement for the fail-closed behaviour
introduced by the envelope.

### Acceptance criteria

- DATO un errore `MlsStateError` con codice stabile `MLS_STATE_*`
  QUANDO la unlock UI lo presenta all'utente
  ALLORA mostra il messaggio mappato per quel codice, mai il raw
  `err.message`.
- DATO un errore `MLS_STATE_OPENMLS_INCOMPATIBLE` con `actions` strutturate
  QUANDO la UI propone il recupero
  ALLORA espone solo le azioni previste (reopen con la build scrivente /
  attendere una build migrante / factory reset esplicito come ultima
  risorsa), nell'ordine di minor distruttività.
- DATO un codice errore non riconosciuto dalla mappa
  QUANDO la UI lo riceve
  ALLORA mostra un messaggio generico sicuro e registra i dettagli tecnici
  solo nei log di sviluppo.
- DATO qualunque percorso di errore della unlock UI
  QUANDO il messaggio viene composto
  ALLORA nessun payload né materiale MLS compare nel testo mostrato
  (proprietà preservata dal contratto della libreria, verificata da test).

## US-002 — Replace spread-based Uint8Array→Base64 conversion on the persistence path

**Adopted from Issue #25** — follow-up of PR #23, residual item 2 of
`docs/security/2026-07-12-review-mls-state-envelope.md`.

`bytesToBase64` (`styx-js/src/utils.js`) uses
`btoa(String.fromCharCode(...bytes))`: the argument spread hits the engine's
argument-count limit (RangeError) far below the envelope's 16 MiB parser cap,
so the practical size limit of `_persistMls` is the helper, not the parser.

To be resolved in Blocco 3 with one of:

- chunked conversion (fixed-size `String.fromCharCode` windows);
- streaming conversion;
- **preferred:** removing Base64 from the persistence path entirely — the
  IndexedDB vault stores binary values natively.

No production change during the spikes.

### Acceptance criteria

- DATO uno stato MLS serializzato di dimensione prossima al cap del parser
  QUANDO `_persistMls` lo persiste
  ALLORA la conversione non solleva `RangeError` per limite di argomenti e
  il limite pratico torna a essere il cap del parser, non l'helper.
- DATO il percorso di persistenza scelto in Blocco 3
  QUANDO la soluzione è integrata
  ALLORA una delle tre opzioni dichiarate è implementata e la scelta è
  registrata (preferita: eliminazione di Base64 con IndexedDB nativo).
- DATO il periodo degli spike
  QUANDO si lavora su questa story
  ALLORA nessun cambiamento raggiunge il percorso di produzione prima della
  decisione di Blocco 3.
- DATO un valore che ecceda comunque i limiti del backend
  QUANDO la persistenza fallisce
  ALLORA il fallimento resta fail-closed e non distruttivo (proprietà della
  story US-004 preservata).

## US-003 — Allowlist for MlsStateError.details fields

**Adopted from Issue #26** — follow-up of PR #23, residual item 1 of
`docs/security/2026-07-12-review-mls-state-envelope.md`.

`details.causeMessage` in `MLS_STATE_MIGRATION_FAILED` /
`MLS_STATE_RESTORE_FAILED` forwards the underlying runtime message and is the
only non-allowlisted channel in the structured errors. The hardened WASM
boundary returns clean errors today (verified by the no-trap tests), but the
contract should be explicit.

Define an allowlist of publishable `details` fields, e.g.: error code;
envelope version; storage schema version; saved and current OpenMLS revision;
saved and current artifact digest; saved and current ciphersuite; suggested
actions.

`causeMessage` must not be propagated automatically: keep the underlying
error for development logs only (or map it to a stable sub-code).

### Acceptance criteria

- DATO un `MlsStateError` con `details`
  QUANDO l'errore attraversa il confine verso la UI o la telemetria
  ALLORA solo i campi dell'allowlist esplicita sono pubblicabili.
- DATO un errore `MLS_STATE_MIGRATION_FAILED` o `MLS_STATE_RESTORE_FAILED`
  QUANDO il runtime sottostante fornisce un messaggio causa
  ALLORA `causeMessage` non è propagato automaticamente: resta nei log di
  sviluppo o è mappato a un sub-codice stabile.
- DATO l'allowlist definita
  QUANDO un nuovo campo viene aggiunto a `details`
  ALLORA il campo è non-pubblicabile finché non è esplicitamente aggiunto
  all'allowlist (default-deny).
- DATO i no-trap test esistenti
  QUANDO l'allowlist è introdotta
  ALLORA i test continuano a passare e un test nuovo asserisce il contratto
  esplicito dell'allowlist.

## US-004 — Document the operational limits of localStorage vs the 16 MiB parser cap

**Adopted from Issue #27** — operational-limit record requested at the
Fase D gate of PR #23.

Facts to state explicitly in the storage documentation (and to keep true in
code):

- `MAX_PAYLOAD_BYTES = 16 MiB` is the **defensive cap of the envelope
  parser**, not a guaranteed capacity of the current backend;
- 16 MiB of binary become >21 MiB in Base64, plus JSON overhead — well
  beyond typical localStorage quotas;
- browsers apply **different quotas** (commonly ~5–10 MB per origin for
  localStorage), so the practical ceiling is browser-dependent and much
  lower than the cap;
- the Blocco 3 IndexedDB vault replaces this operational limit (binary
  values, larger quotas, `navigator.storage.estimate()`);
- a quota error must remain **fail-closed and non-destructive**: today
  `QuotaExceededError` during migration keeps the legacy value + backup
  intact and raises `MLS_STATE_MIGRATION_FAILED` (tested); any future write
  path must preserve this property.

### Acceptance criteria

- DATO la documentazione dello storage
  QUANDO la story è completa
  ALLORA i cinque fatti elencati sopra sono dichiarati esplicitamente e
  coerenti con il codice.
- DATO un `QuotaExceededError` durante la migrazione
  QUANDO il fallimento avviene
  ALLORA il valore legacy e il backup restano intatti e viene sollevato
  `MLS_STATE_MIGRATION_FAILED` (proprietà già testata, da preservare).
- DATO un futuro percorso di scrittura dello storage
  QUANDO viene introdotto
  ALLORA la proprietà fail-closed e non distruttiva su quota è preservata e
  la documentazione resta vera.

## US-005 — Vault IndexedDB engine inside the worker

**Sprint 2, from plan section B3.4 (PR‑4)** of
`docs/superpowers/plans/2026-07-12-styx-vault-implementation-plan.md` —
the plan remains canonical; if this story and the plan disagree, the plan
wins and this file must be corrected.

`src/storage/vault-db.js`, living inside the vault worker (PR #39 runtime):
IndexedDB schema v1 with the ten frozen stores (`meta`, `identity`,
`contacts`, `messages`, `mls`, `outbox`, `push`, `settings`, `migrations`,
`canary` — list frozen in plan §B3.0.1), multi-store transactions resolved on
`oncomplete`, `durability: 'strict'` where supported, auto-close on
`versionchange`, bounded retry on blocked opens (`VAULT_BLOCKED`, 50 ms
backoff), quota mapped to `VAULT_QUOTA_EXCEEDED` fail-closed, bounded
`persist()`, fail-closed upgrades through a migrator registry, destroy with
`onblocked` handled, single-tab access through the existing Web Lock.
**Synthetic records and fixtures only**: every database name is prefixed
`styx-vault-test-*`; no product data exists at this stage.

Depends on the merged PR #39 worker runtime (plan PR‑3): the engine lives
inside the worker. Rollback class **R0** (test-only databases, plain revert).
Gate at merge: the transactional semantics become **frozen** (plan B3.4) —
later stories build on them unchanged.

### Acceptance criteria

- DATO il porting a produzione delle probe P1–P12 dello spike IndexedDB
  QUANDO la suite gira in CI su Chromium e Firefox
  ALLORA tutte le probe sono verdi, incluse kill mid-transaction
  (all-or-nothing), upgrade abort→retry, blocked/versionchange, quota,
  elezione/steal multi-tab e il record binario da 8 MB.
- DATO un crash della pagina durante una transazione multi-store
  QUANDO il database viene riaperto
  ALLORA lo stato è consistente: o tutti gli effetti della transazione o
  nessuno.
- DATO un upgrade di schema che fallisce
  QUANDO la open successiva riparte
  ALLORA il fallimento è fail-closed e ripetibile, senza stato intermedio
  persistito.
- DATO qualunque test di questa story
  QUANDO crea o distrugge database
  ALLORA usa solo nomi prefissati `styx-vault-test-*` e record sintetici,
  mai dati di prodotto.

## US-006 — Lifecycle of a new empty vault

**Sprint 2, from plan section B3.5 (PR‑5)** of the vault implementation
plan (canonical, wins on disagreement). Depends on US-005.

`src/storage/vault.js` inside the worker: the spec §3 state machine with
`CREATE_VAULT / UNLOCK / LOCK / STATUS / CHANGE_PASSWORD / REWRAP /
DESTROY`. Hard constraints: the Root Key is 32 random bytes generated in the
worker, never derived from the password, never persisted in cleartext, never
crossing the worker boundary; the KEK comes only from validated Argon2id;
re-wrap follows spec §7.2 (at least one valid wrapper at every instant,
`rewrapPending`); wrong password answers `VAULT_WRONG_PASSWORD`
non-destructively and without any oracle beyond wrapper well-formedness
(§16.8); an **incompatible** wrapper (well-formed but of an unsupported
version or parameters) fails closed with the structured recovery actions,
still without any oracle beyond its — public — form; LOCK is best-effort
wipe, strong cancellation is terminate; **no migration from localStorage**
in this story. Introduces
`src/config/vault-stage.js` (the `styx.vault.stage` flag, plan §B3.0.6) and
revises the PR‑3 anti-bundle test accordingly — the gate-revision point
recorded in US-001/US-002.

Depends on US-005 and, through it, on the full plan chain PR‑1…PR‑4 (KDF
artifact, crypto formats, worker runtime, IndexedDB engine). This story is
the first **real** use of the persisted wrapper v1 **and manifest v1**
formats. Rollback class **R1**: flag off restores current behaviour; dev
vaults are removed with DESTROY.

**Irreversible-contract gate (plan §16.13, decision n. 1):** from the first
merged vault that persists them, wrapper v1, manifest v1 and record v1
become contracts. The human merge gate of this story is where that decision
is taken.

### Acceptance criteria

- DATO l'insieme delle transizioni della state machine §3
  QUANDO la suite le esercita tutte, valide e vietate
  ALLORA ogni transizione valida riesce e ogni transizione vietata fallisce
  fail-closed con l'errore tipizzato atteso.
- DATO un crash durante il re-wrap, in OGNI punto della sequenza §7.2
  QUANDO il vault viene riaperto
  ALLORA esiste sempre almeno un wrapper funzionante e l'apertura riesce.
- DATO una password errata ripetuta
  QUANDO l'unwrap fallisce su un wrapper ben formato
  ALLORA la risposta è sempre `VAULT_WRONG_PASSWORD`, senza side-effect e
  senza distinzione osservabile dal caso wrapper-corrotto-ma-ben-formato;
  `VAULT_WRAPPER_INVALID` compare solo per forma invalida PRIMA della
  derivazione.
- DATO un wrapper ben formato ma incompatibile (versione o parametri non
  supportati)
  QUANDO il vault tenta l'apertura
  ALLORA il fallimento è fail-closed, non distruttivo, e l'errore porta le
  azioni di recupero strutturate — senza alcun oracle oltre la forma, che è
  comunque pubblica (§16.8).
- DATO il flag `styx.vault.stage` in configurazione developer-only
  QUANDO il vault viene creato, sbloccato e distrutto
  ALLORA il ciclo completo funziona con zero dati di prodotto e il flag off
  riporta l'app al comportamento attuale (rollback R1).
- DATO il bundle di produzione della chat
  QUANDO il flag è off
  ALLORA il test anti-bundle aggiornato dimostra che nessun modulo vault
  raggiunge `dist/`.

## US-007 — Canary namespace end-to-end

**Sprint 2, from plan section B3.6 (PR‑6)** of the vault implementation
plan (canonical, wins on disagreement). Depends on US-006.

The `canary` namespace holds synthetic records generated locally — never
user information — and is exercised end-to-end from the app behind the
`styx.vault.stage` flag: encryption, AAD, persistence, reopen, wrong
password, hand-made bit-flip corruption on the database, crash, re-wrap,
password change, reset, a trial v1→v2 schema upgrade on the canary store
only, service-worker update while UNLOCKED (plan RK8), and simulated storage
eviction (external destroy of the database).

Rollback class **R1** (flag off; canary data is synthetic and disposable).
Gate at merge (plan B3.6): **only from this point onward** may the following
stories touch real product data — everything up to and including the canary
runs on synthetic records exclusively.

### Acceptance criteria

- DATO la matrice §13 della spec vault
  QUANDO la suite canary gira in CI
  ALLORA ogni riga della matrice è coperta ed è verde sul solo namespace
  canary.
- DATO un record canary cifrato corrotto a mano (bit-flip)
  QUANDO il vault lo rilegge
  ALLORA il fallimento è tipizzato, fail-closed e non distruttivo per gli
  altri record.
- DATO l'upgrade di prova v1→v2
  QUANDO viene eseguito
  ALLORA tocca esclusivamente lo store `canary` e resta reversibile.
- DATO qualunque record scritto da questa story
  QUANDO se ne ispeziona il contenuto
  ALLORA è sintetico e generato localmente, mai derivato da informazioni
  dell'utente.
